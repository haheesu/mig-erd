# 데이터 모델링 학습 발표 참고

## 1. 발표 목적

데이터 통합 솔루션을 예로 들어 요구사항이 엔터티, 속성, 관계로 바뀌는 과정을 설명한다. 완성된 ERD를 외우는 것이 아니라 다음 질문에 답하는 것이 목표다.

- 어떤 데이터를 별도 테이블로 분리해야 하는가?
- 설정 데이터와 실행 데이터는 왜 나누는가?
- 1:N 관계는 업무 흐름을 어떻게 표현하는가?
- 단순한 학습 모델은 실제 운영 모델로 어떻게 확장되는가?

참고 산출물:

- `final-data-integration-platform.erd`
- `데이터 통합 솔루션 ERD 테이블 역할 정리.md`

## 2. 권장 발표 구성

### 1장. 문제 정의

RDB, 파일, API, Kafka 데이터를 읽어 MongoDB에 적재하려면 연결정보, 작업 설정, 매핑, 실행 결과를 관리해야 한다.

발표 질문: “화면의 커넥션”은 단순 접속정보인가, 반복 실행 가능한 작업인가?

### 2장. 두 초기 모델의 장점 결합

| 초기 모델 | 가져온 장점 |
|---|---|
| `presentation-simple-data-integration.erd` | 범용 연결정보, 단순한 작업 흐름, 증분 체크포인트 |
| `zium-mongo-migration.erd` | MongoDB 인덱스, 적재 후 쿼리, 상세한 MongoDB 적재 개념 |

최종 모델은 범용성은 첫 번째 모델에서, MongoDB 적재 기능은 두 번째 모델에서 가져왔다. 사용자·권한·알림·DLQ 같은 운영 기능은 핵심 ERD 밖의 확장 후보로 남겼다.

### 3장. 요구사항에서 엔터티 찾기

| 요구 문장 | 도출한 엔터티 |
|---|---|
| Source와 MongoDB 연결을 등록한다 | `data_connection` |
| 연결을 선택해 동기화 작업을 만든다 | `sync_job` |
| 수집 대상과 컬럼을 선택한다 | `source_dataset` |
| Source 필드를 MongoDB 필드로 변환한다 | `field_mapping` |
| 마지막 성공 위치부터 다시 읽는다 | `sync_checkpoint` |
| 실행 결과와 로그를 확인한다 | `job_execution`, `execution_log` |

명사라고 모두 테이블로 만들지 않는다. 반복 저장되고 독립적으로 식별·조회·변경되어야 하는 업무 대상을 중심으로 엔터티를 선택한다.

### 4장. 핵심 ERD 흐름

```text
data_connection
       │ Source / Target
       ▼
    sync_job ─────▶ source_dataset
       │           field_mapping
       │           sync_checkpoint
       │           target_index
       │           post_load_action
       ▼
 job_execution ───▶ execution_log
       
    sync_job ─────▶ audit_history
```

중심 엔터티는 `sync_job`이다. 작업은 연결·대상·매핑을 묶는 설정이고, `job_execution`은 그 설정을 특정 시점에 실행한 사건이다.

### 5장. 연결정보와 작업을 분리한 이유

하나의 MySQL 연결을 여러 동기화 작업에서 사용할 수 있다. 연결정보를 작업마다 복사하면 Endpoint나 인증정보가 바뀔 때 여러 행을 수정해야 한다.

- `data_connection`: 어디에 어떻게 연결하는가
- `sync_job`: 어떤 연결을 사용해 무엇을 언제 동기화하는가

이 분리는 중복을 줄이고 변경 영향을 작게 만든다.

### 6장. 설정과 상태를 분리한 이유

- `source_dataset.incremental_key`: 어떤 기준으로 변경분을 찾는가
- `sync_checkpoint.cursor_value`: 실제로 어디까지 성공했는가

예를 들어 증분키가 `updated_at`이고 체크포인트가 `2026-07-21T10:00:00+09:00`일 수 있다. 전자는 설정이고 후자는 실행할 때 변하는 상태다.

### 7장. 1:N 관계 이해

대표 관계:

- 연결정보 1:N 작업
- 작업 1:N 수집 대상
- 작업 1:N 필드 매핑
- 작업 1:N 실행
- 실행 1:N 로그

부모 한 건에 자식이 여러 건 생기는 업무 규칙을 FK로 표현한다. 예를 들어 작업 정의는 하나지만 실행 이력은 실행할 때마다 계속 추가된다.

### 8장. JSON을 사용한 이유와 한계

RDB, API, Kafka는 필요한 설정이 서로 다르다. 학습 모델에서는 `options_json`, `selected_columns_json`, `transform_json`으로 유형별 차이를 단순하게 수용한다.

장점은 테이블 수를 줄이고 확장하기 쉽다는 것이다. 단점은 DB가 JSON 내부의 타입, 필수값, FK를 직접 보장하기 어렵다는 것이다. 자주 검색하거나 관계 무결성이 필요한 값은 일반 컬럼 또는 별도 테이블로 승격해야 한다.

### 9장. 실행·로그·감사를 분리한 이유

| 테이블 | 답하는 질문 |
|---|---|
| `job_execution` | 실행이 성공했는가? 몇 건 처리했는가? |
| `execution_log` | 처리 중 어떤 기술 메시지가 발생했는가? |
| `audit_history` | 누가 작업 설정이나 상태를 변경했는가? |

세 데이터는 조회 목적과 보존 기간이 다르므로 하나의 이력 테이블로 합치지 않는다.

### 10장. 단순 모델에서 운영 모델로 확장

현재 ERD는 기본 흐름을 학습하기 위한 10개 테이블이다. 실제 구현에서는 요구사항에 따라 다음 순서로 확장할 수 있다.

1. 안정성: `retry_policy`, `dead_letter_record`, `idempotency_record`
2. 보안: `app_user`, `app_role`, `job_permission`, `oauth_token`
3. 운영: `alert_rule`, `notification_event`, `retention_policy`
4. 대규모 운영: `batch`, `job_version`, `schema_change_event`

발표에서는 “왜 빠졌는가”보다 “어떤 요구가 생기면 분리하는가”를 설명한다.

## 3. 발표 시 강조할 트레이드오프

- 테이블이 적으면 흐름은 잘 보이지만 여러 책임이 섞일 수 있다.
- 테이블을 세분화하면 무결성이 좋아지지만 JOIN과 구현 난도가 증가한다.
- 범용 모델은 확장성이 좋지만 유형별 제약을 별도로 검증해야 한다.
- 실행 최신 상태를 작업 테이블에 함께 저장하면 조회는 쉽지만 실행 이력과 불일치할 수 있다.
- 좋은 학습 모델은 모든 기능을 담는 모델이 아니라 핵심 관계와 확장 방향이 설명되는 모델이다.

## 4. 발표 데모 시나리오

1. MySQL Source와 MongoDB Target을 `data_connection`에 등록한다.
2. `sync_job`에서 증분·Cron 작업을 생성한다.
3. `source_dataset`에서 `customer` 테이블과 `updated_at` 증분키를 선택한다.
4. `field_mapping`에서 `customer_name`을 `profile.name`으로 매핑한다.
5. 작업 실행 시 `job_execution`을 생성하고 처리 건수를 갱신한다.
6. 오류가 발생하면 `execution_log`에 기록한다.
7. 성공하면 `sync_checkpoint`의 마지막 처리 시각을 갱신한다.
8. MongoDB 인덱스와 후처리 작업을 적용한다.

## 5. 학습 확인 질문

- `source_connection_id`와 `target_connection_id`가 같은 테이블을 참조하는 이유는 무엇인가?
- `sync_job`과 `job_execution`을 합치면 어떤 중복이 발생하는가?
- 증분키와 체크포인트를 같은 컬럼으로 관리하면 어떤 문제가 생기는가?
- 어떤 JSON 컬럼을 먼저 별도 테이블로 분리해야 하는가?
- 실행 실패와 개별 레코드 실패는 어떻게 다르게 모델링해야 하는가?

## 6. 마무리 메시지

이 모델의 핵심은 `연결 설정`, `동기화 설정`, `실행 결과`를 분리한 것이다. 먼저 단순한 1:N 관계로 기본 흐름을 이해하고, 안정성·보안·운영 요구사항이 구체화될 때 필요한 테이블을 단계적으로 추가하는 것이 데이터 모델링 학습에 효과적이다.
